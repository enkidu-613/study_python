## dialog-58 WebSocket 文档机制与接收循环修正

- 日期：2026-06-14
- 阶段：websocket-realtime
- 类型：dialog
- 主题：WebSocket、FastAPI、Swagger、OpenAPI、`/docs`、`APIWebSocketRoute`、`iter_text()`、`receive_text()`、异步迭代

### 本次学习内容

用户询问：为什么项目里的 WebSocket 没有出现在 `/docs` 文档里，以及应该如何修复。

排查后确认：

- `routers/ws_router.py` 中的 `@router.websocket("/message")` 已经成功注册到 FastAPI 应用。
- 应用路由表中可以看到 `APIWebSocketRoute /ws/message`。
- 但 `app.openapi()` 中没有 `/ws/message`。
- 根因不是路由没注册，而是 FastAPI 的 `/docs` 来自 OpenAPI schema；OpenAPI 主要描述 HTTP 请求/响应接口，不默认展示 WebSocket 协议路由。

修复方式：

- 保留真实 WebSocket 路由 `/ws/message`。
- 新增一个 HTTP 说明接口 `/ws/message-docs`，让 Swagger `/docs` 中能看到 WebSocket 的连接方式、协议类型和示例。
- 修正 WebSocket 标签描述，避免把 WebSocket 误写成 SSE。
- 修正路由注册注释，避免把 `ws_router` 标成 JWT 认证。

随后用户确认理解：Swagger 不能展示 ws 协议路由。

### `iter_text()` 与文档写法的区别

用户追问：为什么把原来的写法改成：

```python
async for message in websocket.iter_text():
```

解释结论：

- 文档常见写法 `while True: message = await websocket.receive_text()` 是每一轮手动接收一条文本消息。
- `async for message in websocket.iter_text()` 是把“持续接收文本消息”的循环交给 Starlette/FastAPI 提供的文本消息迭代器。
- 两者都是推荐写法，区别主要是表达风格：`receive_text()` 更适合初学者看清每次接收，`iter_text()` 更简洁并明确表示只接收文本。
- 原来的 `async for message in websocket:` 不够标准，因为 `websocket` 对象本身不是明确的文本消息迭代器；应使用 `iter_text()`、`iter_json()` 或 `iter_bytes()` 这类明确 API。

### 项目变更

- `routers/ws_router.py`
  - 新增 `GET /ws/message-docs` 说明接口。
  - 将接收循环改为 `websocket.iter_text()`。
- `main.py`
  - 引入并注册 `ws_router`。
  - 修正 WebSocket tag 描述。
  - 修正路由注释。

### 验证结果

OpenAPI 验证：

```text

/ws/message in docs: False
APIRoute /ws/message-docs {'GET'}
APIWebSocketRoute /ws/message None
```

WebSocket 实际连接测试：

```text
你说的是你好
谢谢使用
```

### 学习结论

本次关键理解：

- `/docs` 看的是 OpenAPI，不等于 FastAPI 的完整路由表。
- WebSocket 路由可以注册成功，但不会默认显示在 Swagger。
- 想让学习说明出现在 `/docs`，可以额外提供一个普通 HTTP 文档接口。
- WebSocket 持续接收文本时，推荐二选一：
  - 初学清晰版：`while True + await websocket.receive_text()`
  - 简洁明确版：`async for message in websocket.iter_text()`

## dialog-62 WebSocket Query 参数、认证关闭码与 Hoppscotch 调试

- 日期：2026-06-14
- 阶段：websocket-realtime
- 类型：dialog
- 主题：WebSocket、FastAPI、Query、Depends、Hoppscotch、认证、close code、`HTTPException`、`websocket.close()`、URL 查询参数

### 本次学习内容

用户围绕 WebSocket 认证示例继续追问：

- `Query(...)` 是做什么的。
- 教程里的 WebSocket token 验证为什么和之前 HTTP/JWT 身份验证写法不同。
- `ws_router.py` 的认证路由函数有没有错误。
- 为什么用 Hoppscotch 测试错误 token 时没有看到 `"token 错误"`，只看到默认断开提示。

### 关键理解

`Query(...)` 的含义：

- `token: str = Query(...)` 表示从 URL 查询字符串中读取必填参数 `token`。
- 对应连接地址是 `ws://127.0.0.1:8000/ws/message-auth?token=123456`。
- `...` 在 FastAPI 中表示这个 query 参数必填。

WebSocket 认证和 HTTP 认证的区别：

- HTTP 接口通常通过 `Authorization: Bearer <jwt>` 和 `Depends(get_current_user)` 认证，失败时抛 `HTTPException(401)`。
- WebSocket 连接在握手阶段从 HTTP 升级到 WebSocket，认证通常在 `accept()` 前通过 Query、Cookie 或子协议完成。
- WebSocket 认证失败不适合抛 `HTTPException`，应该用 `await websocket.close(code=4001, reason="token 错误")` 后 `return`。

Hoppscotch 调试点：

- 用户测试地址中出现了拼写错误：`?toekn=33333`，正确参数名应为 `?token=33333`。
- 写成 `toekn` 时，FastAPI 会认为缺少必填参数 `token`，关闭码是 `1008`，reason 是字段缺失。
- 写成 `token=33333` 时，代码会进入自定义认证逻辑，关闭码是 `4001`，reason 是 `token 错误`。
- Hoppscotch 不一定把 WebSocket close reason 当作普通消息展示，可能只显示“发生了一些错误/断开连接”。

### 项目检查与修复

检查 `routers/ws_router.py` 后发现认证路由原先写了：

```python
raise HTTPException(status_code=401, detail="token 错误")
```

这适合 HTTP 路由，但不适合 WebSocket 路由。已改为：

```python
if token != "123456":
    await websocket.close(code=4001, reason="token 错误")
    return
```

### 验证结果

使用 `TestClient` 验证：

```text
/ws/message-auth?token=123456
你说的是测试
谢谢使用

/ws/message-auth?token=33333
WebSocketDisconnect code=4001 reason=token 错误

/ws/message-auth?toekn=33333
WebSocketDisconnect code=1008 reason=Field required
```

### 学习结论

本次关键理解：

- `Query(...)` 是从 URL 的 `?key=value` 中取参数。
- WebSocket 认证失败要用 WebSocket close code，不要用 HTTP 的 `HTTPException`。
- close reason 不等于普通消息，Hoppscotch 可能不会显示。
- 如果想在 Hoppscotch 消息区看到错误内容，可以先 `accept()`，再 `send_json({"type": "error", ...})`，最后 `close()`；但正式认证更推荐在 `accept()` 前拒绝连接。
