## dialog-63：WebSocket 生产部署深度问答与 AI 打断机制

- 日期：2026-06-13
- 阶段：websocket-realtime
- 类型：dialog
- 主题：Redis、Postgres、Kafka、WebSocket、ConnectionManager、asyncio、Task、cancel、CancelledError、broadcast、内存泄漏、多 worker 部署

### 本次学习内容

用户围绕 WebSocket 生产部署和 AI 流式打断机制展开一系列深度追问。

#### Redis / Postgres / Kafka 三剑客概念

- **Redis**：内存数据结构存储，纳秒级 KV 缓存/临时状态，不适合主持久化。
- **Postgres**：对象关系型数据库（ORDBMS），ACID 事务 + 复杂查询，权威数据持久化。
- **Kafka**：分布式事件流平台，高吞吐日志管道，用于系统间异步解耦和流处理。

三者核心差异：Redis 加速、Postgres 持久、Kafka 传输。

#### ConnectionManager 多 worker 限制

`ConnectionManager.active_connections` 是进程内内存列表。多 worker（`--workers 4`）时，每个进程有独立列表，广播只能发到本 worker 的客户端。生产环境需用 Redis Pub/Sub 做跨进程广播。

#### 广播时僵尸连接问题

断开连接后不从列表移除，会导致：
1. 下次 `broadcast` 对死连接 `send_text()` 抛异常，循环中断。
2. 列表只增不减，内存泄漏。

健壮写法是在 `broadcast` 里加 `try/except` 并清理死连接，而非仅依赖 `disconnect()`。

#### AI 打断机制（asyncio 取消协议）

`stream_ai_reply` 内部没有 `stop` 参数，停止靠 asyncio 的任务取消：

```python
current_task = asyncio.create_task(stream_ai_reply(...))
# 用户发 stop
current_task.cancel()
```

`create_task` 把协程扔到后台跑，主循环继续监听消息；返回的 Task 对象就是遥控器。

`cancel()` 不是立刻枪毙，而是在任务内部的**下一个 await 点**自动抛出 `CancelledError`。`stream_ai_reply` 内部显式 `except CancelledError: raise` 是为了让取消信号穿透回到调用者，否则外层 `await current_task` 会永远挂住。

`current_task.done()` 是安全检查：任务已经结束了（正常跑完/已取消/抛异常），就不再执行无意义的 `cancel()`。

#### 关键代码地图

- `routers/ws_router.py#L56-L68` — `ConnectionManager` 内存列表实现
- `routers/ws_router.py#L21-L54` — `stream_ai_reply` 流式 AI 生成 + 取消响应
- `routers/ws_router.py#L112-L144` — `/ai-interrupt` 主循环，处理 `chat` / `stop` / `exit`
- `routers/ws_router.py#L130-L144` — `stop` 分支：cancel → await → 吞异常 → 发 stopped
- `routers/ws_router.py#L169-L171` — `create_task` 启动后台 AI 生成

### 学习结论

- `ConnectionManager` 是单进程原型，生产多 worker 必须换 Redis Pub/Sub。
- `asyncio.create_task` + `task.cancel()` + `CancelledError` 是 Python 异步任务中断的标准 trio。
- `except CancelledError: raise` 看起来多余，实则是让取消协议闭环的关键接力。
- `done()` 防止对已完成任务鞭尸，是防御性编程。
