### 本次学习内容

用户继续 Alembic 实操，遇到更多真实开发中的迁移问题，并完整走通了一次"删库重来"的流程。

#### `revsion` 拼写错误

用户输入 `poetry run alembic revsion --autogenerate -m "add user email"` 报错 `invalid choice: 'revsion'`。正确命令是 `revision`，不是 `revsion`。

#### 已有数据时新增 NOT NULL 字段失败

在 `users` 表中已有数据的前提下，生成新增 `email` 字段的迁移并运行 `upgrade head`，报错：

```
sqlite3.OperationalError: Cannot add a NOT NULL column with default value NULL
```

原因：SQLite 给已有记录加 `nullable=False` 的列时，必须提供数据库层面的默认值 `server_default`。

#### `default=""` 与 `server_default=""` 的区别

- `default=""` 是 SQLAlchemy ORM 层面的默认值。
- `server_default=""` 才是数据库层面的默认值，Alembic 的 autogenerate 不会自动从 ORM 的 `default` 推断出 `server_default`。

修复方式：手动在迁移脚本的 `sa.Column(...)` 里加上 `server_default=""`。

#### SQLite 不支持 ALTER COLUMN TYPE

修复默认值后再次 `upgrade head`，又报错：

```
sqlite3.OperationalError: near "ALTER": syntax error
[SQL: ALTER TABLE users ALTER COLUMN username TYPE VARCHAR(100)]
```

SQLite 不支持 `ALTER COLUMN ... TYPE`，因为本次迁移里同时把 `username` 从 `VARCHAR(50)` 改成了 `VARCHAR(100)`。

#### 开发环境删库重来完整流程

由于前面已有空迁移和失败迁移，且开发环境数据不重要，决定删库重来，生成一份干净的初始迁移：

```bash
# 1. 删除数据库
rm my_database.db

# 2. 删除现有迁移脚本
rm alembic/versions/a6220f34feae_init_tables.py
rm alembic/versions/e25b07889b06_add_user_email.py

# 3. 重新生成完整初始迁移
poetry run alembic revision --autogenerate -m "init"

# 4. 应用到新库
poetry run alembic upgrade head

# 5. 验证表结构
sqlite3 my_database.db ".tables"
sqlite3 my_database.db ".schema users"
```

结果：成功生成 `f21e75ca74bd_init.py`，包含 `users`、`todos`、`documents`、`document_chunks`、`revoked_tokens` 等所有表的完整 `CREATE TABLE` 和索引；`users.username` 为 `VARCHAR(100)`，`email` 为 `VARCHAR(50)`，结构正确。

#### Alembic 前进/回滚命令速查

| 操作 | 命令 |
|------|------|
| 前进到最新 | `poetry run alembic upgrade head` |
| 前进 1 步 | `poetry run alembic upgrade +1` |
| 回滚 1 步 | `poetry run alembic downgrade -1` |
| 回滚到指定版本 | `poetry run alembic downgrade <revision>` |
| 回滚到起点 | `poetry run alembic downgrade base` |
| 查看当前版本 | `poetry run alembic current` |
| 查看历史 | `poetry run alembic history` |

### 学习结论

- SQLite 对 `ALTER TABLE` 支持有限：不能加无默认值的 NOT NULL 列，不能改列类型。
- ORM 的 `default` 不等于数据库的 `server_default`，Alembic 不会自动转换。
- 开发环境遇到迁移死结时，"删库重来"是最快的解决方式；生产环境要用 `server_default` + 分步迁移或 `batch_alter_table`。
- `alembic upgrade head` / `downgrade -1` / `current` / `history` 是日常最常用的四条命令。
