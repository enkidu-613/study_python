**核心教学内容**:

### bcrypt.checkpw 密码验证原理
- bcrypt hash 结构：`$2b$10$` + SALT(22字符) + HASH(31字符) = 60 字符
- 验证流程：从存储 hash 提取 salt → 用同一 salt 重新 hash 用户输入 → 对比两个 hash 值
- **不是解密**（hash 单向不可逆），也**不是分别提取盐**——盐本身就嵌在 hash 字符串中
- 盐来自 `bcrypt.gensalt(rounds=10)` 注册时自动生成加密安全随机盐
- 无需额外盐字段，所有验证信息（版本、轮数、盐、hash）在一个字符串中
- 即使两人密码相同，hash 也不同（盐随机）

### FastAPI Depends 深度解析（三层理解）

| 层 | 理解 |
|----|------|
| **语法糖层** | 省去手动调用，框架自动获取资源注入参数 |
| **多米诺骨牌层** | `Depends` 可以链式嵌套，FastAPI 自动解析依赖图按拓扑顺序执行：`my_route → Depends(get_current_user) → Depends(security) + Depends(get_db)` |
| **三种形态** | `return`（计算/校验，如 get_current_user）、`yield`+`finally`（资源清理，如 get_db）、`Depends(checker)`（闭包工厂，如 require_role） |

## 对话 54: JWT 黑名单机制与 Token 验证深度问答

**时间**: 2026-06-12
**用户任务**:
1. 了解项目黑名单是否有自我过期销毁机制
2. 对比 Token 白名单与黑名单方案
3. 理解 jwt.decode() 的检验能力与边界
4. 理解 JWT_ALGORITHM 的来源与 alg:none 攻击防护
5. 理解过期 Token 的处理流程与黑名单设计目的
6. 理解 token_hash 的 SHA256 机制与 [:64] 切片

**核心教学内容**:

### 黑名单过期机制
- 当前项目 `RevokedToken` 表有 `expires_at` 字段（登出时写入 Token 自然过期时间），但代码中没有定时清理任务
- 黑名单表会只增不减，长期运行积累无用数据
- 生产环境建议用 Redis（自带 TTL）或 APScheduler 定时清理

### 白名单 vs 黑名单
- **黑名单**（业界主流）：只记录已注销 Token，存储 O(注销次数)，正常用户不占空间
- **白名单**：只记录当前有效 Token，存储 O(活跃会话数)，多设备登录时爆炸
- 生产进阶方案：长短 Token（Refresh Token），注销只拉黑 Refresh Token，Access Token 靠短有效期自然失效

### jwt.decode() 检验边界
- 能：验签（HMAC-SHA256 比对）、验过期时间（exp）、验生效时间（nbf）、算法白名单
- 不能：查黑名单、查用户是否被禁用
- `verify_exp=False` 的用途：过期 Token 也能调用 `/logout` 成功登出

### JWT 算法与攻击防护
- `HS256` 是 RFC 7519 标准名称，PyJWT 库内部实现（调用 Python hmac/hashlib）
- `algorithms` 参数是白名单，防止 `alg:none` 攻击（攻击者发送无签名 Token 绕过验证）
- Token Header 自报 `alg`，但库只认用户显式声明的算法列表

### 过期 Token 流程
- 正常请求：查黑名单通过 → jwt.decode 验签 → exp 过期 → 抛 `ExpiredSignatureError` → 401
- 过期 Token 不走进黑名单（验签失败直接返回），因为它本身已是一张废纸

### 黑名单设计目的
- JWT 天生无状态，服务端不记住 Token，不支持注销
- 黑名单是"有状态补丁"：在 Token 仍在有效期内时，提前使其失效（用户主动登出）

### token_hash 与 SHA256
- `_hash_token()` 对完整 Token 做 SHA256，取 `.hexdigest()[:64]`
- SHA256 输出 256bit = 32字节 = 64 个十六进制字符，`[:64]` 恰好取全部
- 存 hash 不存原件：防拖库泄露、定长省空间、单向不可逆

**关键文件**:
- `routers/auth_router.py` — `_hash_token()` / `get_current_user()` / `logout()`
- `models.py` — `RevokedToken` 表结构
- `md/17_JWT用户认证.md` — 黑名单/验签/登出逻辑教学

## 对话 55: SHA-256/黑名单代码清理/身份验证机制全面问答

**时间**: 2026-06-12
**用户任务**:
1. 理解 SHA-256 hexdigest 中 bit/byte/十六进制字符的关系
2. 去除代码中误导性的 [:64] 截取，改为完整哈希
3. 理解静态 vs 动态身份验证、角色存代码 vs 数据库的业界做法
4. 深入闭包在 FastAPI 守卫中的作用及为什么必须用 Depends
5. 理解 ORM 对象与字典的区别、类与对象的关系
6. 理解登出为何不需要 get_current_user、SHA-256 不可逆性
7. 理解 JWT 无状态理念与黑名单的权衡
8. 理解全局异常处理器的作用和 TAGS_METADATA 空标签问题

**核心教学内容**:

### SHA-256 hexdigest 单位关系
- SHA-256 输出 256 bit = 32 字节 = 64 个十六进制字符
- 1 字节 = 8 bit = 2 个十六进制字符
- hexdigest() 是编码(二进制->十六进制字符串)，不是运算
- 原代码 [:64] 恰好取完整哈希但写法有误导性，改为 hexdigest() 直接返回

### 黑名单验证流程
- 验证时：取请求 Token -> 重新 SHA-256 计算 -> 查库比对 hash
- 不需要反推 hash(单向不可逆)，只需重新算一遍比一比
- 确定性(同 Token 同 hash) + 单向性(无法还原) + 碰撞抵抗(不同 Token 不会撞)

### 静态 vs 动态身份验证
- 静态: Token 内含 role，解码即用无额外查询
- 动态: Token 只带 user_id，每次查库，权限实时生效更安全
- 建议: 当前项目用静态，等需后台配置角色时再改动态

### 角色存代码 vs 存数据库
- 小项目写死(USER/ADMIN 字符串)完全够用
- 大项目上 RBAC 多表设计但复杂度高
- 建议: 现在写死，等运营需要后台配置角色时再迁移

### 闭包在 require_role 中的必要性
- FastAPI 的 Depends() 不支持传自定义参数
- 闭包工厂模式: require_role("ADMIN") 动态生成已配置好角色的 Depends 对象
- 不用闭包就要为每种角色组合写单独函数

### Depends 不可替代的原因
- Depends 告诉 FastAPI: "这不是普通默认值，是依赖，请求来时执行它"
- 不用 Depends: 函数默认值是函数对象本身，不会被调用
- 不用 Depends 的后果: 每个接口手写验票 -> 代码爆炸/维护地狱/易漏步骤

### ORM 对象 vs 字典 vs 类与对象
- ORM 对象(User 实例)不是 dict，但属性访问语法相似(user.role vs user["role"])
- Python 中几乎所有东西都是对象(包括 dict/str/int)
- class 是模板，object 是模板造出来的实物
- db.refresh(user) 从数据库拉取最新数据更新 ORM 对象

### 登出为什么不用 get_current_user
- 登出只需要 Token 本身，不需要验证谁/是否过期/什么角色
- get_current_user 会查黑名单和验签，对登出反而是阻碍
- 登出设置 verify_exp=False，允许过期 Token 也能挂失

### JWT 无状态理念与黑名单权衡
- JWT 理想态: 服务端不记任何状态，验签即放行，易扩容速度快
- 黑名单是"必要补丁": 解决 JWT 天生不支持注销的问题
- 权衡: 放弃无状态优雅换取用户主动登出的刚需能力
- 优化方向: Access Token 极短(15min)不存黑名单 + Refresh Token 长存 Redis

### 全局异常处理器与 TAGS_METADATA
- 全局异常处理器兜底处理未被捕获的异常，返回统一 JSON
- 没有时前端可能收到带源码堆栈的 HTML 页面
- 部分 TAGS_METADATA 标签对应路由不存在，需清理

**关键文件**:
- routers/auth_router.py — _hash_token() 去掉 [:64]、闭包守卫、黑名单流程
- models.py — RevokedToken.token_hash 注释修正
- main.py — 全局异常处理器、TAGS_METADATA
