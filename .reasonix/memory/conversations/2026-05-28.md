**时间**: 2026-05-28

2. 理解 ChromaDB 和 documents 表的连接方式
3. 为 `collection.add()` 参数添加详细注释

**核心教学内容**:

### 1. 双存储架构概念
- **类比**: 图书馆系统（SQLite=藏书仓库，ChromaDB=索引卡片柜）
- **为什么需要双存储**: ChromaDB 碎片太小不够回答问题，需要 SQLite 回查完整文档
- **数据流**: 存入时两边各存一份（SQLite存全文+切片，ChromaDB存向量），查询时 ChromaDB 快速找 → SQLite 取详情

### 2. 两条"绳子"连接机制
- **绳子① `embedding_id`**: SQLite 和 ChromaDB 用相同字符串ID（如 `"doc1_chunk0"`）
- **绳子② `metadatas`**: ChromaDB 查询返回时直接带 `document_id` 和 `chunk_index`，无需解析字符串
- **关键澄清**: `metadatas` 标签是手动构造的，不是通过ID查出来的

### 3. `collection.add()` 四个参数详解
| 参数 | 类型 | 参与计算？ | 作用 |
|------|------|-----------|------|
| `ids` | `list[str]` | ❌ | 唯一标识，绳子① |
| `embeddings` | `list[list[float]]` | ✅ 唯一 | 语义相似度计算 |
| `documents` | `list[str]` | ❌ | 展示用原始文本 |
| `metadatas` | `list[dict]` | ❌ | 绳子②，存 document_id |

### 4. 创建文件

- 创建 `dual_storage_demo.py`：完整的双存储演示代码（存入+检索）

## 对话 19: 余弦距离与语义检索细节 Q&A

**时间**: 2026-05-28
**用户任务**: 多轮追问理解语义检索的细节机制

**核心 Q&A**:

### Q1: 语义搜索的整体链路是怎样的？
```
用户问题 → get_embedding() → 向量 → ChromaDB 检索 → 
返回 metadatas（含 document_id）→ SQLite 回查 → 完整文档
```

### Q2: `collection.query()` 的 `include` 参数
- `include=["metadatas", "distances"]` 指定只返回哪些字段
- `ids` 和 `distances` 总是返回，`documents`/`embeddings`/`metadatas` 需显式声明
- 当前代码不需要 `documents`，因为从 SQLite 回查时能拿到完整内容

### Q3: `similarity = 1 - distance` 是什么？
- ChromaDB 返回的 `distance` 是余弦距离（越小越相似）
- `1 - distance` 翻转成余弦相似度（越大越相似），纯粹给人看的
- 余弦距离 = 1 - 余弦相似度（数学定义）
- 所以 `1 - distance` = `1 - (1 - cos(θ))` = `cos(θ)`，兜了一圈还原回余弦相似度

### Q4: 余弦相似度和余弦距离是否相加等于 1？
- 是的，这是数学定义：余弦距离 = 1 - 余弦相似度
- 所以两者相加恒等于 1
- 实际 Embedding 都是正数，余弦相似度 0~1，余弦距离也 0~1

### Q5: ChromaDB 何时计算余弦距离？
- `collection.query()` 内部自动计算（用户看不到）
- 对每个存储向量算 `1 - cos(θ, query_vec)`，排序后取最小的 N 个返回
- 用户代码只负责 `1 - distance` 翻转展示

**用户理解确认**:
- 语义搜索链路理解正确 ✅
- 余弦距离 vs 余弦相似度关系理解正确 ✅
- `include` 参数作用理解正确 ✅

## 对话 21: FastAPI + Chroma 最小原型

**时间**: 2026-05-28
**用户任务**: 进入第12步，将双存储架构封装为 FastAPI API

**核心教学内容**:

### 1. 为什么需要 API 化？
- 脚本只能命令行执行，API 让前端/其他服务能调用
- ChromaDB 从内存模式升级为持久化模式（`PersistentClient`）

### 2. 接口设计（RESTful）
```
POST /rag/documents    → 存入文档（双存储）
POST /rag/search       → 语义检索（双存储协作）
```

### 3. 代码结构
- `routers/rag_router.py`：RAG 业务逻辑（新增）
- `main.py`：注册路由（修改）
- `chroma_db/`：持久化数据目录（自动生成）

### 4. 关键变化
- **持久化 ChromaDB**：`chromadb.PersistentClient(path="./chroma_db")`
- **依赖注入**：`db: Session = Depends(get_db)` 自动管理连接
- **Pydantic 模型**：`DocumentIn`、`SearchIn`、`SearchResult` 定义请求/响应格式

### 5. 创建文件
- `routers/rag_router.py`：完整的 RAG API 路由
- `md/12_FastAPI_Chroma_最小原型.md`：教学文档
