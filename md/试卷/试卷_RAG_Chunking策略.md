## 1. 一句话理解（4 分）

用一句话解释：RAG 里的 Chunking 是什么？它解决什么问题？

**答题区：**


chunk是被剪切出来的语义单位，用于rag数据库的检索 chunk将长文本剪切为几百字符的小片段，避免了超长文本截断报错，并且如果是每小段文本转向量的话会提高搜索的精准度

---

## 2. 参数边界题（5 分）

分别解释下面两个参数：

1. `chunk_size`
2. `chunk_overlap`

并说明为什么不能只设置 `chunk_size`，完全不要 overlap。

**答题区：**

若overlap设置为0，文本会在切割处被硬性切断，会导致语义破碎 和 指代 破碎，总的来说一段上下文有关联的段落或者语句会被不连贯，也会导致模型引用的时候出问题

---

## 3. 场景判断题：切太大和切太小（5 分）

判断下面两种情况分别会造成什么问题：

1. 一个 chunk 里混了注册、价格、退款、数据库迁移四种主题。
2. 一句完整的退款规则被切成三段，每段都只有半句话。

**答题区：**

太大会让模型输出有歧义

太小资料不完整会让模型输出的信息有缺失不完整

## 7. metadata / document / id 边界（5 分）

在向量库里，一条 chunk 记录通常可能包含：

- `id`
- `document` / `page_content`
- `metadata`

再说明：如果要按用户权限或文档来源过滤，应该优先依赖哪一类信息？

**答题区：**

id：是这条数据在rag数据库里的id
document：经过向量化的chunk
metadata：一般存储关联的sql信息和chunk原文，以及一些用户权限和文档来源信息

需要metadata的信息，因为上面睡了，用户权限和文档来源比如sql的信息一般都在这里，或者说文档来源就在这里的title里

### 第 7 题修正版

```text
id = 这条 chunk 记录的唯一编号
document / page_content = 被送进 Embedding 模型的原文正文
embedding vector = document / page_content 被模型转换后的向量
metadata = 跟着正文保存的标签，比如 document_id、chunk_index、title、source、权限信息
```

最重要的一句：

```text
被向量化的是 page_content/document 的文本，不是 metadata；向量本体通常由向量库保存，不是你手写在 metadata 里。
```
