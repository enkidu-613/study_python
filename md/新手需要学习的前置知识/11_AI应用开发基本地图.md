# 11. AI 应用开发基本地图

> **一句话理解：AI 应用开发不是“让模型替你做一切”，而是用代码约束输入、补充资料、调用工具、评估结果并把服务可靠交付。**

## 本章学什么，不学什么

学：模型 API、Prompt、Embedding、RAG、Agent、微调与模型训练的边界，以及如何进入本项目主线。

不学：立刻安装 LangChain、写 Agent 或微调模型；先知道每个部件解决什么问题。

## 准确术语

| 术语 | 准确含义 |
| --- | --- |
| 模型 API | 通过网络请求调用已部署模型的接口。 |
| Prompt | 本次请求随模型输入发送的指令和上下文。 |
| Token | 模型读取和生成文本时使用的文本单位。 |
| Embedding | 把文本转换成数字向量，便于按语义相近程度检索。 |
| RAG | 先检索相关资料，再把资料作为本次上下文交给模型生成回答。 |
| Agent | 模型在受控代码流程中选择并调用工具来完成多步任务。 |
| 微调 | 用示例继续训练已有模型，使某种行为更稳定。 |
| 模型训练 | 从数据学习或显著改变模型参数的过程；不是本路线主线。 |

## 最小模板：一个 AI 应用真正包含什么

```text
用户输入问题
  → 后端校验输入长度、权限和格式
  → 可选：从知识库检索相关资料（RAG）
  → 调用模型 API，附上 Prompt 和资料
  → 校验模型输出的格式
  → 保存必要日志并返回给用户
```

模型只是其中一个组件。安全、数据、错误处理和界面也由你的代码负责。

## 三种问题，选三种工具

| 需求 | 优先选择 | 原因 |
| --- | --- | --- |
| “这次回答请用 JSON 返回” | Prompt + 输出校验 | 要求只影响本次调用。 |
| “回答必须依据今天更新的公司文档” | RAG | 文档能更新、能追溯来源。 |
| “把固定格式的工单稳定分到 30 类” | 先评估 Prompt；必要时微调 | 有大量稳定示例时才考虑长期改变行为。 |

## 常见坑

- 把模型输出当可信程序结果：模型输出需要格式校验、权限控制和错误处理。
- 用微调代替会变化的知识库：资料更新时通常优先 RAG。
- 为了“智能”直接给 Agent 所有权限：工具必须有白名单、参数限制和日志。
- 只看回答是否流畅：还要评估是否正确、是否引用对资料、是否成本可控。
