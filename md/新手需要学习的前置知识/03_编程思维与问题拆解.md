# 03. 编程思维与问题拆解

> **一句话理解：程序不是“猜需求”，而是把明确的输入按明确步骤变成明确的输出。**

## 本章学什么，不学什么

学：输入、处理、输出（IPO）和把大需求拆成可验证小步骤。

不学：复杂架构图或任何 AI 框架。

## 准确术语

| 术语 | 准确含义 |
| --- | --- |
| 输入（Input） | 程序拿到的原始信息，例如用户问题。 |
| 处理（Process） | 程序按照规则执行的步骤。 |
| 输出（Output） | 程序产生的结果，例如分类文字。 |
| 条件 | “如果……就……”的判断规则。 |
| 算法 | 为完成任务而写出的有限、明确步骤；这里不是刷题。 |

## 最小模板：先写流程，再写代码

需求：用户输入一个分数，显示是否及格。

```text
输入：分数
处理：判断分数是否大于等于 60
输出："及格" 或 "需要复习"
```

对应的 Python 形状：

```python
score = 80

if score >= 60:
    print("及格")
else:
    print("需要复习")
```

`score` 是变量，保存输入数据；`if` 根据条件只执行其中一条分支。

## 用四个问题拆需求

面对“做一个课程资料问答工具”，先问：

1. 输入是什么？例如用户问题和资料文件。
2. 输出是什么？例如答案和资料来源。
3. 中间要做哪些事？例如读取资料、查找相关段落、生成回答。
4. 怎么验证？例如用一条已知答案的问题测试。

## 常见坑

- 一开始就写几十行代码：先写 IPO，范围会立刻清楚。
- 输入与输出没定义：程序无法被判断是否正确。
- 只测试“正常情况”：还要问空输入和错误输入怎么办。

## 检查点

- [ ] 能用 IPO 描述“把摄氏温度换成华氏温度”。
- [ ] 能说明 `if` 在上面例子中解决什么问题。
- [ ] 能为一个小工具写出至少一个可验证例子。
