# 08. 互联网、HTTP 与 JSON

> **一句话理解：AI 应用的网页或客户端用 HTTP 把 JSON 请求交给后端，后端处理后再用 HTTP 返回 JSON。**

## 本章学什么，不学什么

学：客户端、服务器、URL、HTTP 请求/响应、方法、状态码与 JSON。

不学：手写前端页面、网络协议底层细节或 WebSocket；后续项目章节会实战它们。

## 准确术语

| 术语 | 准确含义 |
| --- | --- |
| 客户端 | 发起请求的程序，例如浏览器、手机 App 或 Python 脚本。 |
| 服务器 | 接收请求、运行后端代码并返回结果的程序。 |
| URL | 资源地址，例如 `https://example.com/api/tasks`。 |
| HTTP 方法 | 请求意图；`GET` 读取，`POST` 创建或提交处理。 |
| 请求体（body） | 随请求发送的数据。 |
| 状态码 | 响应的结果类别，例如 `200` 成功、`404` 未找到。 |
| JSON | 文本数据格式，使用对象、数组、字符串、数字、布尔值和 `null`。 |

## 最小模板：一次 API 请求与响应

客户端发送：

```http
POST /api/summarize HTTP/1.1
Content-Type: application/json

{"text": "今天学习了 HTTP"}
```

服务器可能返回：

```json
{
  "summary": "用户学习了 HTTP",
  "source": "demo"
}
```

这里 `POST` 是方法，`/api/summarize` 是路径，JSON 是请求体；响应中的 `summary` 与 `source` 是 JSON 对象的键。

## 数据流

```text
浏览器点击“总结”
  → 浏览器发送 POST + JSON
  → FastAPI 后端校验并处理 text
  → 后端调用模型或普通函数
  → 返回 200 + JSON
  → 浏览器显示 summary
```

## 常用状态码

| 状态码 | 含义 | 新手阶段的理解 |
| --- | --- | --- |
| 200 | 成功 | 请求被正常处理。 |
| 201 | 创建成功 | 新记录被创建。 |
| 400 | 请求不正确 | 输入格式或参数不符合要求。 |
| 401 | 未认证 | 尚未登录或令牌无效。 |
| 404 | 未找到 | URL 或数据不存在。 |
| 500 | 服务器错误 | 后端代码或服务发生未处理错误。 |

## 常见坑

- 把 JSON 当 Python dict：看起来相像，但 JSON 的 `true`/`false`/`null` 与 Python 的 `True`/`False`/`None` 不同。
- 用 `GET` 修改数据：读取和修改的意图应清晰区分。
- 看到 500 就改前端：500 通常需要去后端日志和 Traceback 找原因。
- 把 URL、域名和服务器混为一谈：URL 是地址，域名帮助找到服务，服务器运行程序。

## 检查点

- [ ] 能说出客户端和服务器在一次请求中的分工。
- [ ] 能指出上面 HTTP 请求的“方法、路径、请求体”。
- [ ] 能解释 `200`、`404`、`500` 的大致区别。
- [ ] 能读懂一个只含字符串与数组的 JSON 对象。
