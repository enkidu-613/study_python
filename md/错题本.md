### ❌ 错题1：两条绳子的底层原因

**题目**: `metadatas` 里存 `{"document_id": 1}` 而不是解析 `ids` 字符串 `"doc1_chunk0"`，除了"方便"还有什么底层原因？

**我的答案**: 只记得和数据库有联系，其余忘了

**正确答案**: 数据库索引是 B+ 树结构，数字比较是 `O(1)` 直接比大小，字符串要逐字符比较 `O(n)`，且字符串索引占用更多存储。`document_id` 是整数主键，查询速度远快于字符串解析。

**掌握度**: ⭐⭐⭐⭐（2026-06-06 复习通过）

**一句话总结**: 数字比字符串查得快，这是数据库底层决定的。

---

### ⚠️ 半对题1：overlap 的作用（第一轮）

**题目**: `split_text(chunk_size=80, overlap=20)` 为什么要设置 overlap？不重叠直接切不行吗？

**我的答案**: 我不明白

**正确答案**: `overlap` 防止句子在切片边界被切断。例："我喜欢吃苹果"切成"我喜欢吃"和"苹果"就断了语义。重叠 20 字保证上下文连贯。但 overlap **不能消灭**边界断句，只能减少，Embedding 模型靠上下文词汇理解语义。

**掌握度**: ⭐⭐ → ⭐⭐⭐⭐（第三轮已掌握）

**一句话总结**: overlap 是用空间换连贯性，越大越连贯但存储翻倍。

### ⚠️ 半对题2：include 参数默认值

**题目**: `collection.query(include=["metadatas", "distances"])` 里，如果不写 `"metadatas"` 会发生什么？

**我的答案**: 不写就返回默认的属性

**正确答案**: `ids` 和 `distances` 总是返回，但 `metadatas`/`documents`/`embeddings` 默认**不返回**！不写 `"metadatas"` 就拿不到 `document_id`，后续回查 SQLite 时 `chroma_results["metadatas"]` 直接 KeyError 报错。

**掌握度**: ⭐⭐ → ⭐⭐⭐⭐（第三轮已掌握）

**一句话总结**: metadatas 不是默认返回的，不写就报错。

### ⚠️ 半对题3：`documents` 参数的必要性

**题目**: `collection.add()` 的 `documents` 参数存的是碎片文本，`metadatas` 里也有 `document_id`。那 `documents` 是不是多余的？能不能去掉？

**我的答案**: 不能去掉，有存在的道理，但不清楚为什么

**正确答案**: `documents` 不是多余的。单用 ChromaDB 时（不搭 SQLite），`documents` 是唯一文本来源，查询时直接返回展示。双存储架构里虽然从 SQLite 取完整内容，但 `documents` 存碎片文本可用于快速预览。去掉后单库场景无法工作。

**掌握度**: ⭐⭐⭐⭐（2026-06-06 复习通过）

**一句话总结**: documents 让 ChromaDB 能独立工作，不依赖 SQLite。

### ❌ 错题1：`create_task` 不 await 吞异常

**题目**: 用 `asyncio.create_task(foo())` 创建了一个任务但没有 `await`，如果 `foo()` 里抛异常会发生什么？

**我的答案**: 异常会被事件循环捕获并忽略，不报错也不影响程序。

**正确答案**: Python 3.8+ 会在 Task 被垃圾回收时打出 `Task exception was never retrieved` 警告，但**不崩溃，不阻止程序退出**。这是一个静默的错误，所以要么 `await` 捕获异常，要么 `add_done_callback` 显式处理。

**掌握度**: ⭐⭐⭐⭐

**一句话总结**: `create_task` 如果不 `await`，异常会变成"幽灵错误"——不崩溃但会静默丢失。

---

### ❌ 错题2：`gather(return_exceptions=True)` 的结果错误检查方式

**题目**: 以下代码检查 `gather` 返回结果的方式有什么问题？

```python
results = await asyncio.gather(*tasks, return_exceptions=True)
for result in results:
    try:
        print(f"成功了: {result}")
    except:
        if isinstance(result, Exception):
            print(f"失败了: {result}")
```

**我的答案**: 没什么问题？try/except 后接 isinstance 检查双重保障。

**正确答案**: 问题巨大！`try/except` 包裹的是 `print(f"成功了: {result}")`——print 永远不会抛异常，所以 `except` 分支永远不会走到。正确写法是去掉 try/except，直接 `isinstance` 判断：

```python
for result in results:
    if isinstance(result, Exception):
        print(f"失败了: {result}")
    else:
        print(f"成功了: {result}")
```

**掌握度**: ⭐⭐⭐⭐

**一句话总结**: `return_exceptions=True` 时异常以值的形式返回，用 `isinstance(result, Exception)` 判断，不是 `try/except`。

### ⚠️ 半对题1：`isinstance` 与 JS `instanceof` 的区别

**题目**: Python 的 `isinstance` 和 JavaScript 的 `instanceof`，是同一个意思吗？

**我的答案**: 不知道这是什么函数。

**正确答案**: 功能等价。`isinstance(obj, Class)`（Python）= `obj instanceof Class`（JavaScript），都用于检查对象是否属于某个类或类型。但注意区分：
- `isinstance` 检查**类型归属**（"它是这个类的孩子吗？"）
- `is` 检查**身份**（"它就是同一个对象吗？"）

**掌握度**: ⭐⭐⭐⭐

**一句话总结**: `isinstance = instanceof`，别和 `is`（身份比较）搞混。

### ⚠️ 半对题2：Future 唤醒机制理解不全

**题目**: `await future` 时，Future 内部具体发生了什么？`set_result()` 之后事件循环怎么唤醒协程？

**我的答案**: 大概知道 set_result 会触发回调，但细节说不清。

**正确答案**: 三步走：① `await future` → 未来对象发现自己还没结果 → **挂起当前协程**，注册一个"醒来喊我"的回调到 Future 上，`yield self` 把控制权交还给事件循环。② 某个时候 `set_result(value)` 被调用 → 设置 `_state = FINISHED` + 存结果 + **触发所有注册的回调**（包括第①步注册的那个）。③ 回调把挂起的协程重新放入事件循环的**就绪队列** → 事件循环下次轮到时继续执行 `await` 后面的代码。

**掌握度**: ⭐⭐⭐⭐

**一句话总结**: `await future` 挂起+注册回调 → `set_result()` 触发回调→事件循环唤醒协程，三个环节缺一不可。

### ⚠️ 半对题3：Task vs Future 核心区别

**题目**: Task 和 Future 到底有什么区别？什么时候用哪个？

**我的答案**: 没想清楚。

**正确答案**: Task **是** Future 的子类（`class Task(Future)`），但多了个核心功能——**Task 会执行协程代码**（`await` 或 `__next__`），而 Future **不执行任何代码**，只做占位符。当你有一个异步操作的结果需要等待时，用 Future（桥接回调式代码）；当你需要并发执行协程时，用 `create_task` 包装成 Task。

**掌握度**: ⭐⭐⭐⭐

**一句话总结**: Task ≈ 执行协程的工人，Future ≈ 不干活的占位牌。"Task 是 Future"但多了干活的能力。
